## 요구사항 정리

### 1. 개발 목적

- 프록시를 통해 실제 HTTP 요청을 수집한다.
- 수집된 요청을 원본과 최대한 동일하게 재전송한다.
- 재전송한 원본 요청을 baseline으로 저장한다.
- 특정 파라미터 하나만 변조하여 SQLi/XSS 탐지에 활용한다.
- 기존 정적 크롤러 기반 추정 방식의 한계를 줄이고, 실제 요청 기반으로 스캔 정확도를 높인다.

전체 흐름은 다음과 같다.
1. 정적 크롤러 → HTML form/input metadata 수집
2. 프록시 → 실제 HTTP request capture
3. 병합 → parameter role annotation
4. Replay → baseline 생성
5. Mutation → 특정 파라미터만 변조ㄴ
6. Analyzer → SQLi/XSS 판정


### 2. 1차 개발 범위

- GET query string 수집
- POST `application/x-www-form-urlencoded` 수집
- Cookie 포함 요청 저장
- 정적 크롤러 FormMeta와 프록시 요청 병합
- 병합 성공한 파라미터만 검사 후보로 생성
- `request_only`, `unknown` 파라미터는 저장만 하고 active scan 제외
- baseline replay 성공한 요청만 active scan 수행
- baseline 요청에서 500 Internal Server Error 발생 시 `scan_skipped` 처리

---

### 3. 1차 개발 제외 범위

- POST `application/json` 변조
- `multipart/form-data` 변조
- `request_only` 파라미터 active scan
- `unknown` 파라미터 active scan
- CSRF token 자동 갱신
- JavaScript endpoint 정적 분석
- Cookie/Header 값 변조
- 인증 우회/권한 상승 테스트

---

### 4. 정보 수집

### 정적 크롤러 역할

- HTML 구조 정보 수집
    - form method
    - form action
    - input name
    - input type
    - input value
    - textarea
    - select
    - submit button
    - hidden input

정적 크롤러는 파라미터의 의미를 추정하는 데 사용한다.

### 프록시 역할

- 실제 전송된 HTTP 요청 수집
    - 실제 method
    - 실제 URL
    - 실제 query/body 값
    - 실제 Cookie
    - 실제 Referer
    - 실제 User-Agent
    - 실제 요청 순서
    - JS 실행 이후 발생한 요청

### 5. 요청 저장 구조

요청은 JSON 형태로 저장한다.

```
{
  "request_id":"req-001",
  "method":"GET",
  "full_url":"http://localhost:8080/vulnerabilities/sqli/?id=1&Submit=Submit",
  "base_url":"http://localhost:8080/vulnerabilities/sqli/",
  "path":"/vulnerabilities/sqli/",
  "parameters": [
    {
      "name":"id",
      "location":"query",
      "value":"1",
      "index":0
    },
    {
      "name":"Submit",
      "location":"query",
      "value":"Submit",
      "index":1
    }
  ],
  "headers": {
    "Cookie":"PHPSESSID=abc123; security=low"
  },
  "body":null,
  "body_type":"query",
  "content_type":null,
  "response_status_code":200,
  "response_body_length":3520,
  "timestamp":"2026-06-09T12:00:00"
}
```

파라미터는 dict가 아니라 list로 저장한다.

같은 이름의 파라미터가 여러 번 나올 수 있기 때문이다.

### 6. Scope 필터

프록시는 모든 요청을 수집할 수 있으므로 target 범위를 제한한다.

저장 대상:

- target host 내부 요청
- GET query 요청
- POST form-urlencoded 요청

제외 대상:

- 외부 도메인
- 이미지 파일
- CSS 파일
- 폰트 파일
- 소스맵 파일
- favicon

제외 확장자:

```
.css
.js
.png
.jpg
.jpeg
.gif
.svg
.ico
.woff
.woff2
.map
```

단, `.js` 파일은 추후 endpoint 분석용으로 사용할 수 있으므로 나중에 옵션으로 저장 가능하게 둔다.

### 7. 중복 요청 제거

같은 구조의 요청은 중복 제거한다.

중복 판단 기준:

```
method + base_url + parameter names + body_type
```

parameter value는 중복 판단에 포함하지 않는다.

예:

```
GET /item?id=1
GET /item?id=2
```

두 요청은 값은 다르지만 구조는 같으므로 같은 스캔 대상으로 본다.

대표 요청 선택 기준:

- 로그인된 요청
- 응답 status가 200인 요청
- 파라미터 값이 비어 있지 않은 요청
- 오류 페이지나 redirect가 아닌 요청

---

### 8. 정적 크롤러와 프록시 요청 병합

병합 기준:

- method 일치
- 정규화된 form action URL과 request base URL 일치
- form input name과 request parameter name이 하나 이상 겹침

### URL 정규화

- form action이 비어 있으면 현재 page_url 사용
- 상대경로 action은 page_url 기준으로 절대 URL 변환
- 프록시 full_url에서는 query string 제거 후 base_url 저장
- URL fragment(`#...`)는 비교 제외
- 맨 뒤 `/`는 제거해서 비교

예:

```
form.action_resolved = http://localhost:8080/vulnerabilities/sqli
request.base_url     = http://localhost:8080/vulnerabilities/sqli
```

---

### 9. 병합 결과 예시

정적 크롤러가 수집한 form:

```
<formmethod="GET"action="">
<inputtype="text"name="id">
<inputtype="submit"name="Submit"value="Submit">
</form>
```

병합 결과:

```
{
  "request_id":"req-001",
  "matched_form_id":"form-001",
  "method":"GET",
  "url":"http://localhost:8080/vulnerabilities/sqli/",
  "parameters": [
    {
      "name":"id",
      "location":"query",
      "value":"1",
      "html_type":"text",
      "origin":"matched_form_input",
      "role":"user_input",
      "scan_eligible":true,
      "priority":"high",
      "reason":"matched text input"
    },
    {
      "name":"Submit",
      "location":"query",
      "value":"Submit",
      "html_type":"submit",
      "origin":"matched_form_input",
      "role":"control",
      "scan_eligible":false,
      "priority":"none",
      "reason":"matched submit input"
    }
  ]
}
```

해석:

- `id`
    - `type=text`로 확인됨
    - 실제 요청값 `1` 확인됨
    - 검사 대상
- `Submit`
    - `type=submit`으로 확인됨
    - control 값
    - 검사 제외

---

### 10. 병합 실패 처리

정적 크롤러 정보와 프록시 요청이 매칭되지 않는 경우가 있다.

예:

- JS 요청
- 동적 요청
- 크롤러가 놓친 form 요청
- form metadata 없이 프록시에서만 확인된 요청

처리 방식:

- 해당 요청은 버리지 않고 `request_only`로 저장한다.
- 현재 개발 범위에서는 `request_only`를 active scan하지 않는다.
- `unknown` 파라미터도 현재 개발 범위에서는 active scan하지 않는다.
- 추후 확장을 위해 `origin`, `role`, `reason`은 기록한다.

예:

```
{
  "name":"Submit",
  "value":"Submit",
  "html_type":null,
  "origin":"request_only",
  "role":"unknown",
  "scan_eligible":false,
  "priority":"none",
  "reason":"request_only parameter skipped in current phase"
}
```

---

### 11. 파라미터 role 분류

- `user_input`
    - 사용자 입력값
    - 검사 대상
    - 예: text, password, search, email, url, number, tel, textarea
- `control`
    - form 동작 제어값
    - 검사 제외
    - 예: submit, button, reset
- `token`
    - CSRF token, nonce 등
    - 검사 제외 또는 token handler 대상
- `session`
    - 세션 또는 인증 관련 값
    - 검사 제외
    - 예: PHPSESSID, JSESSIONID
- `hidden_input`
    - hidden input이지만 token은 아닌 값
    - 조건부 검사
- `choice_input`
    - select, radio, checkbox
    - 조건부 검사
- `request_only`
    - 프록시 요청에서만 확인된 값
    - 현재 개발 범위에서는 검사 제외
- `unknown`
    - 역할 확정 불가
    - 현재 개발 범위에서는 검사 제외

### 12. 재전송 요구사항

재전송 모듈은 캡처된 요청을 원본과 최대한 동일하게 재전송한다.

입력 정보:

- method
- url
- headers
- query parameter
- body
- content-type
- cookie

동작:

1. 저장된 요청을 읽는다.
2. 불필요하거나 재계산해야 하는 헤더를 제거한다.
3. method, url, param, body, header를 복원한다.
4. 요청을 전송한다.
5. 응답 status_code, body, length, hash, elapsed time을 저장한다.

제거할 헤더:

- Host
- Content-Length
- Connection
- Accept-Encoding

유지할 헤더:

- Cookie
- User-Agent
- Referer
- Content-Type

---

### 13. Baseline 저장

active scan 전에 원본 요청을 먼저 replay한다.

- 원본 응답을 baseline으로 저장한다.
- baseline 없이 SQLi/XSS를 취약하다고 판단하지 않는다.

baseline 저장 정보:

- status_code
- response_body
- response_length
- response_hash
- elapsed_time
- final_url
- redirect_count
- content_type

baseline 실패 시:

- active scan 대상 제외
- `scan_skipped` 기록
- finding 생성 안 함

---

### 14. 500 Internal Server Error 처리

### baseline 요청에서 500 발생

- 원본 요청 자체가 정상 재현되지 않은 것으로 본다.
- active scan 대상에서 제외한다.
- finding을 생성하지 않는다.

```
baseline → 500
→ scan_skipped
→ finding 생성 안 함
```

### attack 요청에서 500 발생

- 무조건 실패로 버리지 않는다.
- SQLi payload 때문에 서버 에러가 발생했을 가능성이 있다.
- 단, attack 500만으로 SQLi confirmed 처리하지 않는다.

```
baseline → 200
attack → 500
→ error-based SQLi evidence 후보
→ DB 에러 시그니처 확인 후 판단
```

### 15. 파라미터 변조 정책

기본 원칙:

- 한 번에 하나의 파라미터만 변조한다.
- 나머지 파라미터는 원본 값을 유지한다.
- Cookie와 세션 관련 헤더는 유지한다.
- token, session, control 값은 변조하지 않는다.
- `request_only`, `unknown` 파라미터는 현재 개발 범위에서는 변조하지 않는다.

예:

```
원본:
id=1&Submit=Submit
```

올바른 변조:

```
id=1' OR '1'='1&Submit=Submit
```

잘못된 변조:

```
id=1' OR '1'='1&Submit=1' OR '1'='1
```

### 16. 변조 대상 / 제외 대상

### 변조 대상

- 정적 크롤러와 프록시 요청이 정상적으로 병합된 파라미터
- `html_type`이 사용자 입력 필드로 확인된 파라미터
- `scan_eligible=true`인 파라미터
- baseline replay가 성공한 요청의 파라미터

확정 검사 대상:

- text
- search
- textarea
- password
- email
- url
- number
- tel

hidden 중 식별자 또는 조회값으로 보이는 값:

- id
- uid
- no
- idx
- page
- category
- user_id

### 제외 대상

- submit
- reset
- button
- csrf
- csrf_token
- `_token`
- nonce
- authenticity_token
- user_token
- session
- PHPSESSID
- JSESSIONID
- request_only
- unknown

### 조건부 검사 대상

- hidden
- select
- checkbox
- radio

조건부 대상 처리:

- token 이름 패턴이면 제외 또는 token handler 대상으로 둔다.
- id, uid, no, page, category 같은 조회값이면 검사 후보로 둔다.
- 판단이 어려우면 현재 개발 범위에서는 제외하고 추후 확장 후보로 기록한다.

### 17. Scan Candidate 생성 기준

Scan Candidate는 실제 active scan에 사용할 최종 후보이다.

생성 대상:

- `origin = matched_form_input`
- `scan_eligible = true`
- `role = user_input` 또는 검사 가능한 `hidden_input`
- baseline replay 성공

제외 대상:

- `role = control`
- `role = token`
- `role = session`
- `origin = request_only`
- `role = unknown`
- baseline replay 실패
- baseline 요청에서 500 발생

예:

```
[
  {
    "request_id":"req-001",
    "target_param":"id",
    "location":"query",
    "base_value":"1",
    "vuln_candidates": ["sqli","xss"],
    "priority":"high"
  }
]
```
